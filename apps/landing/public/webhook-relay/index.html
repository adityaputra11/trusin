WEBHOOK RELAY, EXPLAINED

A webhook relay receives an event once and delivers it with context.

A relay sits between a webhook provider and your service. It can acknowledge the provider quickly, persist the event, route it to a target, and preserve a timeline when the target is unavailable.

Webhook relay vs. queue

A queue gives workers durable work to process later. A webhook relay is the operational layer around that queue: it accepts HTTP requests, applies routing, attempts delivery, handles retryable failures, and shows the outcome to developers.

NeedQueue aloneWebhook relay
Receive provider HTTP requestsRequires an ingest applicationProvides a dedicated ingest endpoint
Route by source or hookApplication code requiredRouting configuration and delivery targets
Investigate a failed deliveryLogs and custom toolingEvent and attempt timeline
Recover after transient failuresWorker implementation requiredRetry policy plus controlled replay

When to use one

Use a relay when a provider webhook is important enough to need visibility and recovery. Typical examples include payment events, account changes, order updates, and deployment notifications. For local development, the trusin CLI streams events to a loopback target without a public tunnel.

Frequently asked questions

What happens after a target responds with an error?

trusin records the attempt and retries retryable failures with exponential backoff. See Webhook retries for the recovery model.

Can a relay prevent duplicate processing?

Network failures can make an attempt result uncertain. Target handlers should use an event ID or idempotency key so repeated delivery is safe.
